Ce qu'un utilisateur va pouvoir faire.

Un seul chatbot — façon ChatGPT/Claude, spécialisé imagerie médicale. Le clinicien-chercheur pointe son dataset et, depuis la conversation, onboarde ses données, entraîne, infère, visualise les résultats, compare, garde et reproduit ses expériences — puis déploie le modèle en privé. Chaque capacité correspond à un outil konfai-mcp réel (56 outils aujourd'hui) : l'UI est la façade, pas un nouveau moteur.

Où ça tourne

Trois topologies pour l'utilisateur

L'interface est toujours un navigateur ; ce qu'il y a derrière change. La donnée ne quitte le site que dans les topologies où le calcul est on-prem ou dans l'onglet.

TopologieCôté userLa donnée…Statut
Tout-localnavigateur + GPU local install une fois (modèle Jupyter)ne quitte jamais la machinesocle prêt
Serveur on-premnavigateur / client mince GPU sur le serveur du laboreste dans le réseau de l'institutiondéjà en prod (konfai-apps + Slicer)
In-browsernavigateur seul WebGPU, aucun serveurne bouge pas du toutPhase 1 (export ONNX)
Ton LLM, ton choix. Le cerveau du chatbot est branchable — le MCP est agnostique : API Claude (meilleure qualité), modèle local (100 % on-prem), ou l'endpoint LLM de l'institution. Point clé privacy : la donnée image ne va jamais au LLM. Les outils MCP tournent en local et ne renvoient que du texte (chemins, stats, métriques, YAML) — le LLM voit la conversation, pas les voxels. Deux plans : le plan donnée (toujours local) et le plan raisonnement (le LLM que tu choisis). Un cerveau cloud n'uploade aucun voxel.
Périmètre & statut honnête

Ce qui marche déjà, ce qui reste

La plupart des capacités sont déjà pilotables sur l'API MCP — le gros du travail est l'UI. Deux vrais chantiers cœur : l'intégration du viewer et l'export ONNX.

Déjà faisable aujourd'hui (via konfai-mcp)
  • Onboarder un dataset, l'inspecter, corriger les alias
  • Les 3 voies : app telle quelle · fine-tune · train from scratch
  • Écrire + valider la config (jusqu'au forward+backward réel)
  • Train / predict / eval / resume / folds, monitoring live
  • Comparer, diff, leaderboard (inter-sessions)
  • Garder / exporter / ré-importer / rejouer un run
  • Empaqueter un modèle en app ; l'exécuter en remote on-prem
À construire
  • L'UI Studio — chatbot React + BFF FastAPI (M0/M1) sur l'API MCP actuelle, zéro code cœur
  • Le viewer NiiVue — overlays 3D dans l'onglet aujourd'hui : aperçu PNG ; NiiVue pas encore branché
  • Export ONNX (Phase 1) — konfai export → model.onnx + manifest.json verrou dur du déploiement in-browser
  • Tiling volume + géométrie côté runtime navigateur
Périmètre à ne pas surpromettre : le déploiement in-browser ne concerne que les modèles feed-forward gelés (segmentation / synthèse). Le registration 3D reste Python (le grid_sample de Burn est 2D). Le spike konfai-rs est GO (parité 5,5e-6 WGPU sur impact_synth 26M), mais la course WebGPU réelle en navigateur et le tiling volume restent à faire — et rien ne tourne in-tab tant que l'export ONNX (Phase 1) n'a pas produit le contrat model.onnx + manifest.json.
