Part I

What KonfAI is today

The foundation the idea rests on. If you already know KonfAI, skip to Part II — but these are the load-bearing facts that make Studio credible.

The framework

A model described in YAML, not written in code

KonfAI (Boussot & Dillenseger, 2025 — arXiv:2508.09823) is a modular, fully-configurable deep-learning framework for medical imaging. A model, its data pipeline, losses/metrics, optimizer, augmentations, and the whole train/predict/evaluate workflow are described in YAML and mapped onto Python objects by a reflection engine — without editing code. Three ideas run through it.

PILLAR 1

Config-by-reflection

The engine reads a callable's signature and builds its arguments from the YAML subtree it owns, recursing into nested objects. Resolved defaults are written back to disk — so a run leaves a fully-resolved, replayable record of the experiment.

the config IS the experiment
PILLAR 2

Lazy, patch-based imaging

Volumes are never loaded whole into RAM. Data is read as overlapping patches (optionally streamed) and predictions are reassembled with overlap blending — so KonfAI runs large 3D volumes on modest hardware.

scales to real clinical data
PILLAR 3

Declarative models

Networks are routed graphs — written as Python classes, or entirely as a .yml via a safe model builder. A shipped catalog covers UNet, VNet, UNETR, ViT and more, addressable by name.

no boilerplate
The workflow

Command-driven: three files, one reproducible workspace

KonfAI is run from the CLI. Four states map to three YAML files, and every run writes a self-contained workspace keyed by a training name — checkpoints, TensorBoard stats, the resolved-config snapshot, predictions, and metric JSON.

CommandFileRoot keyPurpose
TRAIN / RESUMEConfig.ymlTrainer:Model + dataset + losses + augmentations + optimizer/schedulers + training params.
PREDICTIONPrediction.ymlPredictor:Load model(s), patch / TTA / ensemble inference, output post-processing.
EVALUATIONEvaluation.ymlEvaluator:Predictions vs ground truth → per-case + aggregate metric JSON.
Why this matters for the idea: because the whole lifecycle is config in → workspace out, an agent can drive it without touching code — it only ever writes YAML, launches subprocess jobs, and reads JSON. That is exactly what makes an agentic front end (and a browser one) possible.
The ecosystem & the proof

A core, three packages around it, and a clinical bridge

KonfAI ships as a small family of independent packages layered on the core's public API. Studio is not built on sand — it reuses these.

clinical integration

Part II

The idea: KonfAI Studio

Think ChatGPT or Claude — but it builds and runs medical-imaging models on your own data, and nothing leaves your machine. Everything in Part I already exists; the idea is to put it behind one chatbot and close the last two links — visualization and in-browser deployment.

The objective

What we're actually trying to achieve

Not another engine. The surface that makes the existing engine usable by a human — and welds three scattered bricks into one product. There are two layers to the goal; both matter.

Picture the whole product as one chatbot — the ChatGPT / Claude experience, specialized for medical-imaging AI. A clinician-researcher opens Studio, points it at their own dataset, and says what they want: “segment this organ,” “generate a synthetic CT.” From that conversation alone, an agent does the ML — inspects the data, writes the config, trains, infers, and shows the results as overlays on their scans. No terminal, no ML engineer.

Every run is kept and reproducible: the resolved config is the experiment, so the researcher builds a personal, replayable library of their work — to compare, re-run, share, or publish. And unlike a cloud assistant, the model runs privately in the browser, offline — their data never leaves the machine.

The shift: konfai-mcp already exposes this whole lifecycle to an agent, but on the command line, reserved for people who read a terminal. Studio is the same engine behind a conversational surface a domain expert can use.

Layer 1 — the mission

Collapse the distance between a clinical question and a deployed, private, reproducible model — from months of ML engineering to a conversation. The person who owns the problem owns the whole loop, on their own data, which never leaves the institution.

Layer 2 — the strategy (Fideus)

Studio is the product surface that turns the open KonfAI engine into a defensible product — likely open-core (open engine, monetizable Studio layer). The moat is the empty market intersection below. This is what makes KonfAI a company, not just a framework.

“Describe the task” → a trained, reproducible model that runs in your browser, offline.
No one else closes this loop. That's the product.
The white space

Three capabilities that, combined, exist nowhere else

Each incumbent covers one, sometimes two. Their intersection is empty on the market — and that is exactly where KonfAI already holds the raw material.

A · AGENTIC AUTHORING

An agent creates the model

It inspects the data, writes the config, trains and iterates. The agentic pipelines published in 2026 stay training-free: they orchestrate already-trained models; they produce none.

B · REPRODUCIBLE

The config IS the experiment

The resolved YAML is a complete, replayable record of the run — a direct answer to the reproducibility crisis (most medical-DL work shares neither code nor config).

C · PRIVATE, ZERO-INSTALL

The model runs client-side

The frozen model runs as a native binary, edge, Slicer, or browser — no Python, no arbitrary code, data doesn't move. No 100%-browser medical segmentation demo exists to date.

KonfAI StudioA, B and C exist separately. No one has united them in a single product — this is the loop no one closes, and the one KonfAI is uniquely able to close.

The honest caveat

Two distinct planes — state them plainly

“Private” doesn't mean the same thing when training and when deploying. Saying so makes the story stronger, not weaker.

Train — drive the MCP

Needs a GPU → “private” = on-prem

Training runs on the lab's workstation or server, not in the tab. Data stays on site, but compute needs a local machine. Already out of reach for MONAI (server + Python + install) and nnU-Net.

Deploy — infer

Genuinely 100% browser

The frozen model runs in the tab (WebGPU/WASM), offline. Here patient data doesn't move at all — the “nothing leaves the machine” story is literal and airtight.

Scope, not to oversell: only frozen feed-forward models become portable (segmentation, synthesis). 3D registration (grid_sample) doesn't run on Burn today and stays Python. That's a real subset of the catalog — but it is precisely the subset that makes the demo that sells.
